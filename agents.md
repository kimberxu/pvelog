# PVE AIOps Agent Guidelines

This document outlines the project structure, development workflows, coding standards, and testing guidelines for the PVE AIOps intelligent monitoring and troubleshooting system.

## Project Structure

The project is divided into two main components: a centralized Python controller and a lightweight Go agent.

- **`controller/`**: Centralized backend application built with Python 3.11+, FastAPI, and LangGraph.
  - `api/`, `core/`, `services/`, `db/`, `scheduler/`: Source code for the backend API, AI log analysis logic, database models, and scheduled tasks.
  - `tests/`: Python test cases.
- **`agent/`**: Lightweight daemon written in Go, deployed directly on Proxmox VE (PVE) host nodes.
  - `cmd/pve-agent/`, `internal/`: Go source code for log collection, pushing, and secure command execution.
  - `tests/`: Go test files.
- **`docs/`**: Project documentation, including deployment guides (`deployment.md`).
- **`scripts/`**: Shell scripts for automating the deployment and installation processes.
- **`docker-compose.yml`**: Docker composition file for running the Controller and its dependencies.

## Build, Test, and Development Commands

### Controller (Python)
- **Run Locally (Docker)**: `docker compose up -d`
  *Builds the Docker image for the controller and runs the FastAPI application using Uvicorn.*
- **Run Locally (Native)**: `cd controller && uvicorn api.main:app --host 0.0.0.0 --port 42791 --reload`
  *Runs the API server natively for local development (requires setting environment variables).*
- **Test**: `pytest controller/tests/`
  *Runs the Python test suite.*

### Agent (Go)
- **Build Locally**: `cd agent && go build -o bin/pve-agent ./cmd/pve-agent`
  *Compiles the Go agent binary.*
- **Install Locally**: `./agent/install_local.sh`
  *Builds and installs the agent on the local system.*
- **Test**: `cd agent && go test ./...`
  *Runs all tests for the Go agent.*
- **Deploy**: `./scripts/deploy_agent.sh`
  *Automated script to deploy the agent to a remote PVE node.*

## Coding Style & Naming Conventions

> 💡 **Note**: This is an AI-assisted "vibe coding" project. Maintain the existing patterns and structures generated by the AI coding assistant.

- **Python (Controller)**:
  - Follow **PEP 8** style guidelines.
  - Use 4 spaces for indentation.
  - Strictly use type hints (`typing` module, Pydantic models) for function signatures and class definitions.
  - Use `snake_case` for variables and functions, `PascalCase` for classes.
- **Go (Agent)**:
  - Adhere to standard Go formatting rules. Run `gofmt` before committing.
  - Use `camelCase` for private variables/functions and `PascalCase` for exported identifiers.
  - Use tabs for indentation as per Go standards.
- **General**: Keep configuration centralized. Update `.env.example` and `README.md` if new environment variables are added.

## Testing Guidelines

- **Frameworks**: 
  - Use `pytest` for the Python controller tests.
  - Use the built-in Go `testing` package for agent tests.
- **Requirements**:
  - Ensure any new tool added to the LangGraph analyzer has corresponding validation tests.
  - Maintain tests for API routes (`api/routes/`) and core logic.
- **Naming Conventions**:
  - Python tests: Name files `test_*.py` and functions `test_*`.
  - Go tests: Name files `*_test.go` and functions `Test*`.

## Commit & Pull Request Guidelines

### Commit Messages
We follow the **Conventional Commits** specification. Ensure your commit messages clearly communicate the intent of the change.
- `feat:` for new features (e.g., `feat: disable check_service_status tool for LLM`).
- `fix:` for bug fixes (e.g., `fix: remove provider-specific fields from assistant message`).
- `chore:` for maintenance tasks, dependencies, etc.
- `refactor:` for code refactoring without adding features or fixing bugs.

### Pull Requests
- Provide a clear, descriptive title.
- Detail the "Why" and "What" of the changes in the PR description.
- If fixing a bug or implementing a feature, reference the related issue number.
- Ensure all automated tests pass (`pytest` and `go test`).
- Include screenshots or terminal output for any UI or significant workflow changes.

## Security & Architecture Overview
- **Zero-Trust Communication**: Agent and Controller communicate securely using a Pre-Shared Key (PSK) with HMAC-SHA256 signature verification.
- **Agent Sandbox Execution**: The Agent runs with minimal privileges and uses an explicit whitelist to execute diagnostic commands securely, preventing arbitrary code execution by the LLM.

## Agent-Specific Instructions
- **Communication Language**: Always communicate with the user in Chinese
- **Planning Mode Workflow**: For any modifications to the project, you must first outline a plan (create an implementation plan artifact), wait for explicit user confirmation, and only proceed to execution after approval
